How the supervisor is wired at startup

Java AgentConfig.java 
// Every sub-agent is an interface — no implementation code. LangChain4j generates proxies.
@Bean
public PulseSupervisorAgent supervisorAgent(
    ChatModel chatModel, BrandFinderAgent brandFinderAgent,
    OfferFinderAgent offerFinderAgent, GeneralQAAgent generalQAAgent,
    AgentMonitor agentMonitor) {

  return AgenticServices
    .supervisorBuilder(PulseSupervisorAgent.class)
    .chatModel(chatModel)
    // 20 slots: supervisor loops multiple planning rounds per request.
    // keyed by A2A taskId — each request gets a fresh, isolated window.
    .chatMemoryProvider(memoryId -> MessageWindowChatMemory.withMaxMessages(20))
    .subAgents(brandFinderAgent, offerFinderAgent, generalQAAgent)
    // AgentMonitor.inheritedBySubagents() = true, so BrandFinder/OfferFinder/GeneralQA
    // are all tracked automatically — zero per-agent wiring needed.
    .listener(agentMonitor)
    .build();
}

// BrandFinderAgent: interface with exactly one @Agent-annotated method
@Bean
public BrandFinderAgent brandFinderAgent(ChatModel chatModel,
    Mem0Client mem0Client, Mem0Properties mem0Properties) {
  return AgenticServices.agentBuilder(BrandFinderAgent.class)
    .chatModel(chatModel)
    .tools(new AgentMemoryTool(mem0Client, mem0Properties,
               AgentMemoryTool.TOPIC_BRANDS, "BrandFinderAgent"))
    // system + user + saveMemory call + result + final AI = 5 slots exactly
    .chatMemoryProvider(memoryId -> MessageWindowChatMemory.withMaxMessages(5))
    .build();
}
Protocol

A2A: why it matters

The Agent-to-Agent protocol (A2A) is Google's open standard for inter-agent communication. By exposing Pulse HK as an A2A endpoint, it becomes a first-class citizen in any multi-agent ecosystem — discoverable via AgentCard, invokable by any A2A-compliant client, and composable into larger agent pipelines without any bespoke integration.

⬡ What A2A gives Pulse HK
🪪 Self-describing identity
The AgentCard advertises name, version, capabilities, skills, and security requirements — any A2A client can auto-discover Pulse HK's interface.
📨 Standardised message envelope
Requests arrive as message/send JSON-RPC calls. RequestContext carries the task ID, message parts, and optional metadata for identity.
🆔 Task lifecycle management
Each call gets an isolated task ID — Pulse HK uses this as the @MemoryId for the supervisor's chat window, ensuring zero cross-request state leakage.
🔐 Identity in metadata
When no HTTP headers are present, A2A's message.metadata carries userId and countryCode — enabling agent-to-agent calls to propagate identity automatically.
http://localhost:3001  ·  A2A UI — sending a message/send to Pulse HK
A2A UI — a React client that auto-discovers the AgentCard and sends message/send JSON-RPC calls. The agent's response arrives as a completed Task with state: completed and an artifacts array.

Key insight: The A2A task ID and the userId serve different purposes. The task ID scopes the supervisor's short-term planning memory — it resets on every call. The userId scopes Mem0 — it accumulates across every call, building the personalisation layer.

Long-Term Memory

The memory loop that makes it personal

The design principle is simple: short-term memory lives in a message window; long-term memory lives in Mem0. The two never mix inside the same buffer. This separation is the single most important architectural decision in the system.

Why inject into the prompt, not into chat history?

An early prototype loaded Mem0 memories as SystemMessage entries and returned them from ChatMemoryStore.getMessages(). It broke badly: the supervisor's planner LLM saw brand names from the user's memory and hallucinated agent names from them — routing a request for Kellogg to an agent called KelloggAgent that didn't exist.

The fix: getMessages() returns an empty list. Memories are prepended as a structured text block at the top of the user's next message — treated as factual context, not instructions.

Topic-scoped metadata — the key to clean retrieval

Every memory written to Mem0 carries two metadata fields: topic and source. At retrieval time, three parallel POST /search calls each filter by one topic bucket, so the enriched prompt renders memories in clean labelled sections.

// A pure Java interface — LangChain4j generates the entire proxy at startup.
// No implementation class exists. The @Agent name pins the agentId so the
// supervisor's planner LLM always routes to this agent deterministically.
public interface BrandFinderAgent {

  @SystemMessage("""
      Analyze the following user request and extract the brand name.

      Steps:
      1. Identify the brand name from the user request.
      2. Call saveMemory ONCE with a concise sentence describing the user's brand interest,
         e.g. "User is interested in Kowloon Dairy products".
      3. Return ONLY the brand name as your final response.
      """)
  @UserMessage("userRequest: {{userRequest}}")
  // name= pins the agentId — must match what supervisorBuilder enumerates in its system prompt.
  @Agent(name = "BrandFinderAgent",
         description = "Extracts the brand name from the user request text")
  String findBrandName(@V("userRequest") String userRequest);
}

MessageWindowChatMemory — sized by contract

The memory window for each agent is sized to the maximum number of messages that agent can ever produce in one invocation. Over-sizing wastes context; under-sizing evicts the system prompt.

Supervisor
max 20 messages
Holds all planning rounds, sub-agent results, replanning cycles. Keyed by task ID — resets each request.
BrandFinder
max 5 messages
system + user + saveMemory call + result + final AI response.
OfferFinder
max 7 messages
system + user + getOfferDetails call/result + saveMemory call/result + final AI.
GeneralQA
max 5 messages
Same budget as BrandFinder — no tool calls for data retrieval.

The AgenticScope bridge: sub-agent interfaces have fixed method signatures — there's nowhere to thread userId through as a parameter. LangChain4j's AgenticScope solves this: CustomAgentExecutor writes identity into the scope at request start; any downstream component reads it without the interface ever changing.

Java AgentMemoryTool.java 
// Each sub-agent gets its own AgentMemoryTool instance wired with a fixed topic + source.
// The @Tool description is what the LLM reads to decide WHEN to call this method.
@Tool("Save a memory fact about the user to long-term storage. "
    + "Call this once after completing your main task. "
    + "Write a concise, factual sentence about what the user was interested in.")
public String saveMemory(
    @P("Concise factual sentence, e.g. 'User is interested in Kowloon Dairy products'")
    String content) {

  // userId resolved from ThreadLocal — NOT a method parameter.
  // This keeps the agent interface signature fixed and testable in isolation.
  String userId = UserContextHolder.get()
      .map(UserContext::userId)
      .orElse(UserContext.DEFAULT_USER_ID);

  var request = new MemoryCreate(
      List.of(new Message("user", content)),
      userId, null, null,
      Map.of(META_TOPIC, topic, META_SOURCE, source),  // metadata tags
      mem0Properties.infer(),  // infer=true: Mem0 runs its own LLM to extract semantic facts
      null, null
  );
  mem0Client.addMemory(request);
  return "Memory saved.";  // returned to LLM, not the user
}
Identity & Context Propagation

One identity, three sources, zero surprises

User identity must reach at least three independent places — the Mem0 read call, the Mem0 write call, and the AgenticScope — without any of those components sharing a call stack. A ThreadLocal carrier solves the synchronous path; AgenticScope covers LangChain4j's async planner loops.

UserContext Resolution — Three-Priority Chain
1
JWT Bearer Token — highest priority
Authorization: Bearer <token> header parsed by UserContextInterceptor.tryJwt(). Decodes the base64url payload, extracts sub as userId and a countryCode claim. Runs in preHandle so identity is available before the A2A handler dispatches.
UserContextHolder.set(new UserContext(sub, countryCode))
2
Gateway Headers — Kong / API gateway proxy
X-User-Id + X-Country-Code headers set by an upstream gateway (e.g. Kong with JWT decode plugin). Resolved by tryGatewayHeaders() when no JWT is present. Common for internal microservice calls where the gateway strips the token.
UserContextHolder.set(new UserContext(xUserId, xCountryCode))
3
A2A message.metadata — agent-to-agent calls
When no HTTP headers are set — common when an AI agent calls Pulse HK as a sub-agent — message.metadata carries {"userId": "...", "countryCode": "..."}. Resolved inside CustomAgentExecutor.execute() via UserContextResolver.resolve(message).
UserContextHolder.set(UserContextResolver.resolve(message))

Dual cleanup: UserContextHolder.clear() is called in both afterCompletion() (interceptor) and the executor's finally block. Defensive double-clearing because Spring Boot's executor may run on a different thread pool than the HTTP handler — without it, a failed request could contaminate the next one.

Java CustomAgentExecutor.java 
@Override
public void execute(RequestContext context, EventQueue eventQueue) throws JSONRPCError {
  TaskUpdater updater = new TaskUpdater(context, eventQueue);
  try {
    if (context.getTask() == null) updater.submit();
    updater.startWork();

    // Resolve identity in priority order:
    // 1. JWT via UserContextInterceptor (already in ThreadLocal)
    // 2. X-User-Id headers via UserContextInterceptor (already in ThreadLocal)
    // 3. message.metadata fallback (set here)
    UserContext userContext = UserContextResolver.resolve(context.getMessage());

    // Scope key = userId: sub-agents read identity without method params
    populateScope(userContext.userId(), userContext);
    UserContextHolder.set(userContext);

    String response;
    try {
      response = customClientExecutorHandler.execute(context);
    } finally {
      // Defensive clear: executor may run on a different thread from the HTTP handler
      UserContextHolder.clear();
    }

    updater.addArtifact(List.of(new TextPart(response)), null, null, null);
    updater.complete();
  } catch (Exception e) {
    throw new JSONRPCError(-32603, "Agent execution failed: " + e.getMessage(), null);
  }
}

private void populateScope(String scopeKey, UserContext ctx) {
  AgenticScope scope = getOrCreateScope(scopeKey);
  scope.writeState(UserContext.KEY_USER_ID,      ctx.userId());
  scope.writeState(UserContext.KEY_COUNTRY_CODE, ctx.countryCode());
}

private AgenticScope getOrCreateScope(String scopeKey) {
  AgenticScope existing = supervisorAgent.getAgenticScope(scopeKey);
  if (existing != null) return existing;
  if (supervisorAgent instanceof AgenticScopeOwner owner)
    return owner.registry().create(scopeKey);
  return DefaultAgenticScope.ephemeralAgenticScope();  // safety net
}
Priority 1: JWT Bearer Priority 2: X-User-Id header Priority 3: A2A message.metadata

ThreadLocal hygiene: UserContextHolder.clear() is called in two places — afterCompletion (the interceptor) and the executor's finally block — because in Spring Boot, the executor may run on a thread from a different pool than the HTTP handler. Defensive double-clearing prevents state leaking across requests on pooled threads.

@GetMapping(value = "/report", produces = MediaType.TEXT_HTML_VALUE)
public String htmlReport() {
  // HtmlReportGenerator is provided by LangChain4j.
  // One endpoint, self-contained HTML — no external dashboard needed.
  return HtmlReportGenerator.generateReport(agentMonitor);
}

@GetMapping(value = "/summary", produces = MediaType.APPLICATION_JSON_VALUE)
public Map<String, Object> summary() {
  Set<Object> ids = agentMonitor.allMemoryIds();
  return Map.of(
    "trackedUsers", ids.size(),
    "users", ids.stream().map(id -> Map.of(
      "userId",     String.valueOf(id),
      "successful", agentMonitor.successfulExecutionsFor(id).size(),
      "failed",     agentMonitor.failedExecutionsFor(id).size(),
      "ongoing",    agentMonitor.ongoingExecutionFor(id) != null ? 1 : 0
      // ongoing > 0 = stuck or in-flight agent — use for health checks
    )).toList()
  );
}

Why bucket by taskId, not userId? The monitor uses the @MemoryId value — which is the A2A task ID — as the bucket key. Each request gets its own execution tree, ideal for waterfall rendering. The summary() endpoint maps back to users by querying successfulExecutionsFor(userId) across all known IDs.

MonitoredExecution buckets: The monitor uses the @MemoryId value as the bucket key. Because Pulse HK sets the task ID (not userId) as @MemoryId, executions are bucketed per-request — ideal for waterfall rendering. The AgentMonitorController's summary() endpoint still maps back to users because all task IDs for one user are queried via successfulExecutionsFor(userId).

Design Decisions

Six decisions that define the architecture

Every non-trivial decision in Pulse HK enforces one of four invariants: clean role separation, correct memory scoping, safe identity propagation, and zero-surprise observability. Each maps to a concrete line of code — not just a principle.

🎯 Task ID as @MemoryId, not userId
The supervisor's chatMemoryProvider is keyed by the A2A task ID. Each request starts with a clean planning window. Two concurrent requests from the same user cannot corrupt each other's in-flight state. Cross-request personalisation is exclusively Mem0's job — a hard boundary enforced in code.
💬 Memories into the prompt, not chat history
An early prototype injected Mem0 memories as SystemMessage entries inside getMessages(). The planner hallucinated agent names from past brand names — routing a request to a fictitious KelloggAgent. Fix: getMessages() always returns List.of(); memories prepend the user message as factual context instead.
🧹 Memory windows sized by contract
Window size = maximum messages the agent produces in one invocation. BrandFinder: 5 (system+user+call+result+AI), OfferFinder: 7 (two tool call/result pairs), Supervisor: 20 (multi-round planning). Deterministic sizing avoids token-budget tuning entirely — no per-model calibration needed.
🔗 ThreadLocal + AgenticScope dual identity
ThreadLocal covers the synchronous interceptor-to-tool path. AgenticScope covers LangChain4j's async planner loops where thread identity isn't guaranteed. Both are set at request start and cleared at end — in two places for defence-in-depth against thread-pool contamination.
🧪 AgenticScopeAccess not SupervisorAgent
LangChain4j's built-in SupervisorAgent carries its own @Agent invoke(String). Our @MemoryId-parameterised overload would trigger a "multiple agent methods" build error at startup. Extending AgenticScopeAccess only avoids this validation while still enabling scope wiring and registry access.
🌊 Topic-segmented Mem0 metadata
Every memory write carries topic + source metadata. Three parallel POST /search calls at retrieval time filter each topic bucket, assembling brand, preference, and general memories in clearly labelled sections — preventing the planner from treating past brand names as agent directives.
